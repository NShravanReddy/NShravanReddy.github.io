## Dataset vs DataFrame

| Dataset | DataFrame |
| --- | --- |
| Distributed collection of **typed objects** partitioned across nodes in a cluster | Distributed collection of **Row objects** (table with rows and columns) |
| **Type-safe** (compile-time checking) | **No type-safe** (errors at runtime) |
| Works with **custom objects (e.g., case classes)** | Works with **Row (generic structure)** |
| Available in **Scala and Java** | Available in **Python, Scala, Java** |
| Slightly harder to use | Easier to use (SQL-like operations) |
| Less commonly used in real-world DE | Widely used (especially in PySpark) |

## Why Spark Replaced Hadoop

- **In-memory computation** — Spark processes data in memory (RAM), making it much faster compared to Hadoop MapReduce, which relies heavily on disk I/O.
- **Lazy evaluation** — Transformations are lazy; they are not executed until an action is called. This allows Spark to optimize the execution plan using a DAG.
- **Fault tolerance** — Spark provides fault tolerance using lineage. If a partition is lost due to failure, it recomputes only the lost data using the DAG, without restarting the entire job.
- **Partitioning (Parallel processing)** — Spark splits data into partitions and distributes them across cluster nodes, enabling parallel processing and faster execution.

When you run an **action**, Spark breaks work like this:

```
Job → Stage → Task
```

## Schema: DDL vs StructType

Define schema additionally to change the data type from source. In Spark, schema defines the structure of data (column names and data types). We define schema explicitly using DDL or StructType to avoid incorrect type inference and to enforce correct data types from the source.

### 1. DDL (SQL-like)

```python
schema = "id INT, name STRING, salary DOUBLE"
df = spark.read.option("header", True).schema(schema).csv("file.csv")
```

- Simple, easy to write
- Not good for complex data

### 2. StructType (Programmatic)

```python
from pyspark.sql.types import *

schema = StructType([
    StructField("id", IntegerType(), True),
    StructField("name", StringType(), True),
    StructField("salary", DoubleType(), True)
])

df = spark.read.option("header", True).schema(schema).csv("file.csv")
```

- Full control
- Supports nested data
- Used in production

### Why not inferSchema?

- Extra scan → slow
- Wrong data types possible
- Not reliable
